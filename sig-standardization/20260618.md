### Bug: recommendations ineffective

sponsor: @mbuechse

- issue [standards#1206](https://github.com/SovereignCloudStack/standards/issues/1206)
- distinguish two concepts:
  1. recommendations in standards (strange beasts)
  2. testcases for recommendations (cluttering the tables)
- testing recommendations is kind of pointless and advertising their results is misleading
  - operator's intention is unknown
  - recommendations can be satisfied by coincidence
  - no guarantee that this will extend into the future
  - --> PR [1223](https://github.com/SovereignCloudStack/standards/pull/1223) (open for review and tentatively deployed) illustrates how much simpler the compliance monitor could become if recommended testcases vanished
- distinguish three types of recommendations:
  - soft: wouldn't ever become a requirement
    - move someplace else, docs, blog, supplement
  - hard: should become a requirement
    - make it one!
  - unclear: authors couldn't decide what they wanted
    - worst case: burden of decision should be with us
    - potentially touching the question of "what product is SCS" really -- involve Forum and Community by offering elaborate proposals
- wish list
  - maybe do not remove recommendations from test suite (include with feature flag on command line)
  - show regions separately in overview
    - maybe each in the SCS-compatible column
- proposed next steps
  - move soft recommendations to supplements (this would also remove the corresponding testcases from reporting) or drop altogether (see item "gratuitous regulation")
  - turn hard recommendations into requirements in next version (this would move the corresponding testcases there)
  - create issues for unclear recommendations, otherwise treat like soft recommendations
    - involve community/forum
    - both synchronously (prepare for regular meetings)
    - and ping "groups of experts" on Github

### FYI: older scope versions removed from overview

sponsor: @mbuechse

- implemented in PR [standards#1205](https://github.com/SovereignCloudStack/standards/pull/1205) (merged!)
- we used to show v3 and v4 (but only when v5.1 was not achieved)
- reasoning was that some customers might want to know that, because their workloads could be stuck on an older version, and compliance with a current version does (in general) not imply compliance with an older version
- however, older versions are not targeted by the operator, and compliance with them is purely coincidental and not reliable
- plus, compliance with older versions could be undesireable for the operators and the ecosystem at large, for these versions sometimes have outdated requirements
  - quite harmless, but instructive example: Ubuntu 20.04

### Scrap approval of test results, introduce color scale instead

sponsor: @mbuechse

- issue [standards#1207](https://github.com/SovereignCloudStack/standards/issues/1207)
- PR [standards#1221](https://github.com/SovereignCloudStack/standards/pull/1221) open for review (tentatively deployed!)
- approving test case results is too labor intensive
- instead, better distinguish between the cases
  1. test case was most recently failed = RED 🛑 if required for a non-draft version, otherwise YELLOW 🟧
  2. test case is missing (no recent result) = RED ❓
  3. test case was most recently inconclusive (test crashed or something) = WHITE ❔
  4. test case was most recently passed = GREEN ✅
- overall result for scope version: carried over from the first one in the enumeration above that applies to any required test case
  1. failed = RED 🛑
  2. missing = YELLOW 🟧
  3. inconclusive = GREEN w/asterisk ✅*
  4. passed = GREEN ✅
- see [this comment](https://github.com/SovereignCloudStack/standards/pull/1221#issuecomment-4734127618) for an illustration of case 3 above
